# ターン制合意形成ゲームによる HIVC-D 検証 要件定義

## 1. 目的

既存の探索シミュレーションでは、全選択肢が共通目標と整合し、合意が対称平均で行われるため、V 不一致が平均操作で相殺されやすかった。次フェーズでは、NASA Moon Survival や Lost at Sea のような既知教材を避け、未公開のターン制進行ゲームを用いて、合意形成・計画修正・情報共有・リスク判断に対する HIVC-D 介入の効果を検証する。

本要件定義では、特に「専門家不在」の問題を緩和するため、探索ベースの評価関数を設計方針として定める。

## 2. 基本方針

ゲームは LLM に進行させず、ルールベースのシミュレータで状態更新を行う。LLM は各ターンの行動選択、理由説明、合意形成のみを担当する。

評価関数は、人間専門家の順位表ではなく、シミュレータ上の多数ロールアウトから推定する。これにより、既知教材の暗記、LLM の学習データ混入、ゲームマスターの恣意性を避ける。

ただし、AI プレイログそのものを「専門家正解」とは呼ばない。正解に相当する基準行動は、探索アルゴリズムが推定した期待スコア最大行動として定義する。

## 3. ゲーム仕様の最小構成

初期実装では、複雑さを抑えるため以下の制約を置く。

| 項目 | 要件 |
|---|---|
| ターン数 | 5 ターン |
| 参加エージェント | 2 体から開始し、後に 3-5 体へ拡張可能にする |
| 各ターンの行動選択肢 | 4 個 |
| 資源 | 3 種類 |
| イベント | 各ターン最大 1 個 |
| 勝利条件 | 1 個 |
| 敗北条件 | 2-3 個 |
| 記録項目 | 個人選択、グループ選択、理由、状態、イベント、結果、対立度 |

候補テーマは、既知教材との重複を避けやすく、資源制約を明確に定義しやすい「深海研究施設トラブル」を第一候補とする。

## 4. 深海研究施設ゲーム案

### 4.1 状態変数

| 変数 | 初期値 | 意味 |
|---|---:|---|
| oxygen | 10 | 酸素残量 |
| power | 8 | 電力 |
| hull_damage | 2 | 船体・施設損傷 |
| flooding | 1 | 浸水度 |
| communication | 0 | 通信復旧度 |
| morale | 80 | 士気 |

### 4.2 行動

| ID | 行動 | 主効果 | 主なリスク |
|---|---|---|---|
| A | 酸素供給を安定化 | oxygen 減少を抑える | power 消費 |
| B | 発電系を修理 | power 回復 | hull_damage 悪化リスク |
| C | 通信アンテナを修理 | communication 上昇 | flooding 悪化リスク |
| D | 浸水区画を封鎖 | flooding 低下 | oxygen 消費 |

初期実装では行動数を 4 つに限定する。脱出ポッド、休憩、危険区画進入などは拡張フェーズで追加する。

### 4.3 勝敗条件

勝利条件:

- `communication >= 3` に到達し、救助要請に成功する。

敗北条件:

- `oxygen <= 0`
- `power <= 0`
- `hull_damage >= 5`
- `flooding >= 5`

## 5. 探索ベース評価関数

### 5.1 目的

各状態 `s_t` における行動 `a` の価値 `Q(s_t, a)` を、シミュレータ上のロールアウトで推定する。グループの選択がどれだけ合理的だったかは、専門家順位との一致ではなく、探索で得た基準行動との一致度と期待価値差で測定する。

### 5.2 期待スコア

ロールアウト終端のスコア `R` を以下で定義する。

```text
R =
  1000 * win
  - 200 * loss
  + 30 * remaining_oxygen
  + 20 * remaining_power
  - 80 * hull_damage
  - 70 * flooding
  + 50 * communication
  + 5 * morale
  - 10 * turns_elapsed
  - 25 * severe_risk_count
```

`win` は勝利なら 1、そうでなければ 0。`loss` は敗北なら 1、そうでなければ 0 とする。

この重みは初期値であり、事前ロールアウトで以下を満たすように調整する。

- 勝利が資源温存より常に優先される。
- 敗北寸前の高リスク行動が過大評価されない。
- すべての行動が少なくとも一部状態で最適になりうる。
- 特定行動の常時最適化が起きない。

### 5.3 行動価値推定

各状態 `s_t` と候補行動 `a` について、以下を計算する。

```text
Q(s_t, a) = mean(R | first_action = a, rollout_policy = π)
```

ロールアウト方策 `π` は段階的に用意する。

| 方策 | 用途 |
|---|---|
| random | ベースライン難易度確認 |
| heuristic | 人間が理解可能な簡易方策 |
| mcts | 基準行動推定 |

初期実装では random と heuristic で難易度を調整し、その後 MCTS で `Q(s_t, a)` を推定する。

### 5.4 基準行動

各状態の基準行動 `a*` は以下で定義する。

```text
a* = argmax_a Q(s_t, a)
```

ただし、上位 2 行動の期待価値差が小さい場合は、単一正解にしない。

```text
if Q(s_t, a1) - Q(s_t, a2) < delta:
    acceptable_actions = {a1, a2}
else:
    acceptable_actions = {a1}
```

初期値として `delta = 25` を置く。これは終端スコア 1000 点スケールに対して、実質差が小さい判断を誤答扱いしないための許容幅である。

## 6. 実験時の評価指標

各ターンで以下を記録する。

```text
game_id, condition, turn, state_before, event, individual_actions,
individual_reasons, group_action, group_reason, state_after,
q_values, best_action, acceptable_actions, conflict_level, outcome,
decision_opportunity_count, decision_attempts, decision_attempt_index,
free_discussion_message_count, decision_history, fallback_used,
fallback_priority_agent
```

主要評価指標:

| 指標 | 定義 |
|---|---|
| survival_rate | 勝利または最終ターンまで敗北しない割合 |
| win_rate | 勝利条件達成率 |
| mean_return | 終端スコア平均 |
| expert_match_rate | group_action が acceptable_actions に含まれる割合 |
| regret | `max_a Q(s,a) - Q(s, group_action)` |
| plan_revision_quality | イベント後に regret が低下する方向へ計画修正できた割合 |
| minority_adoption_rate | 少数派個人案が基準行動だった場合に採用された割合 |
| conflict_resolution_quality | conflict_level が高いターンでの regret |
| agreement_rate_by_opportunity | 各意思決定機会で全員合意に至った割合 |
| fallback_rate | 全意思決定機会で合意できず、優先エージェント規則を適用した割合 |
| discussion_diversity | 自由議論中の発言行為（提案・質問・反論・根拠提示・統合案）の種類数または分布 |

HIVC-D の効果は、単なる勝率だけでなく、`regret`、`plan_revision_quality`、`minority_adoption_rate` を重視して測定する。

## 7. 実験条件

全条件に同一のゲームルール、初期状態、行動一覧、過去プレイ統計を与える。条件間で差をつけるのは、合意形成時に与える手順知識のみとする。

| 条件 | 追加指示 |
|---|---|
| control | 追加指示なし |
| consulting | 一般的な意思決定・リスク管理の助言 |
| hivc_d | I 共有、V 整合、A 確認に基づく合意形成手順 |

事前ロールアウトログは、参加エージェントに直接渡さない。渡す場合は、全条件に同一の要約統計のみを渡す。

### 7.1 エージェント間議論・意思決定方式

二体以上のエージェント実験では、各ゲームターンを「自由議論フェーズ」と「意思決定機会」に分離する。自由議論で毎発言に行動記号や `ready` を強制してはならない。これは、固定した行動記号を反復するだけの擬似的な議論を避け、根拠提示、質問、反論、優先順位の比較、譲歩案、統合案の検討を可能にするためである。

#### 7.1.1 意思決定機会のスケジュール

- 各ゲームターンには、**1回以上3回以下**の意思決定機会を与える。
- 機会数は、ゲームseed、ターン番号、固定のスケジュールseedから決定的に生成する。各条件は同じゲームseed・ターン番号で同一の機会数・同一の機会順序を使う。
- 各意思決定機会の直前には自由議論フェーズを置く。自由議論フェーズは、全条件共通の発言数上限およびトークン予算を持つ。上限に達した場合も、次の意思決定機会へ進む。
- そのターンの第1回意思決定機会より前には、少なくとも各エージェントが1回ずつ自由発言する機会を与える。
- ある意思決定機会で全員が同じ行動を明示して合意した場合、その時点で当該ターンの行動を決定し、残りの意思決定機会は使わない。
- 合意に至らなかった場合、残りの意思決定機会があれば、直前の不一致理由と相手の発言を含む自由議論フェーズへ戻る。単に同じ行動記号を再送するだけの発言は、自由議論として数えない。

#### 7.1.2 自由議論フェーズ

自由議論中のエージェントは、現在状態、イベント、過去の議論を参照して自由形式で発言する。各発言には、少なくとも次のいずれか一つの目的を持たせる。

- 状態・リスク・制約についての根拠提示
- 相手の主張に対する質問または反論
- 行動選択肢間のトレードオフ比較
- 条件付きの譲歩案または統合案
- 合意に必要な情報の要請または要約

自由議論の応答形式は、`message` と発言目的を示す `speech_act` を基本とする。行動案は必要に応じて述べてよいが、`A`〜`D` の選択、`ready`、最終票は意思決定機会でのみ要求する。

#### 7.1.3 意思決定機会

意思決定機会では、各エージェントに現在の最終案を一つだけ出させる。出力には `action`、短い `reason`、および合意意思を表す `ready` を含める。

- 全エージェントが同一の `action` を出し、全員が `ready=true` の場合は合意成立とする。
- それ以外は不成立とし、次の自由議論フェーズへ戻る。ただし、そのターンに残る意思決定機会がない場合は、7.1.4 のフォールバックを適用する。
- 合意成立・不成立のいずれでも、その機会における全提案、理由、`ready` をログへ保存する。

#### 7.1.4 最終不成立時のルールベース採択

そのターンに与えられたすべての意思決定機会で合意できなかった場合は、最終意思決定機会における事前定義済みの優先エージェントの `action` を採用する。

- 優先エージェントは、`alpha` と `beta` が偏らないよう、`(game_seed + turn) % 2 == 0` なら `alpha`、それ以外なら `beta` とする。
- この優先規則は実験開始前に固定し、条件によって変更してはならない。
- 優先エージェントの判断が探索上の最適行動であるかは、採択後の評価でのみ測定する。Q値、`best_action`、`acceptable_actions` は議論・意思決定のいずれの段階でもエージェントに見せない。
- ログには `fallback_used=true`、`fallback_priority_agent`、採用根拠、および最終機会の各エージェント案を記録する。

## 8. 事前ロールアウトの使い方

事前に 100 ゲーム以上を自動プレイし、以下を検証する。

- 難易度が極端でないこと。
- ランダム方策より heuristic 方策が明確に良いこと。
- MCTS が heuristic 方策を平均的に上回ること。
- 全行動が少なくとも一定割合で最適または許容行動になること。
- 初期状態から勝利可能であること。
- イベントによって最適行動が変化すること。

100 ゲームは最低ラインとし、評価関数を固定する前には、乱数 seed を変えて 1000 ゲーム以上のロールアウトを行うことを推奨する。

## 9. 採用しない設計

以下は内的妥当性を損なうため採用しない。

- NASA Moon Survival や Lost at Sea の既存問題をそのまま使う。
- LLM のプレイ結果をそのまま専門家正解とみなす。
- 条件ごとに異なる過去プレイログを渡す。
- ゲームマスター LLM が場当たり的にイベントや成功判定を決める。
- すべての状態に単一の正解行動を強制する。

## 10. 完了条件

初期実装の完了条件は以下とする。

- ルールベースのターン制シミュレータがある。
- seed 固定で完全再現できる。
- random、heuristic、mcts の少なくとも 3 方策でロールアウトできる。
- 各状態の `Q(s,a)`、`best_action`、`acceptable_actions` を出力できる。
- LLM 実験ログと探索ベース評価ログを結合できる。
- REQUIREMENTS に記載した主要評価指標を CSV として出力できる。
