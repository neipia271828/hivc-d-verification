# 評価妥当性修正 実装計画

## 1. 目的と対象

2026-07-13 のレビューで見つかった次の二件を修正し、複数ゲーム実験の集計値と役割別情報の非対称性を有効に保つ。

1. `plan_revision_quality` と `route_switch_quality` がゲーム境界をまたいで前ターンを参照する。
2. beta の「発進条件充足」診断が、非公開の `flooding` を二値的に推測可能にする。

本計画は実装前の作業分解であり、ゲームルール、勝利条件、評価関数の重み、既存の CSV スキーマは変更しない。

## 2. 完了条件

- 時系列依存の両指標は `condition` と `seed`（なければ `game_id`）の組単位でのみ前ターンを参照する。
- 入力行の並び順や、別ゲームの同じ `turn` の存在によって両指標が変化しない。
- beta に与える状態表示・役割固有情報は、`flooding` または `hull_damage` だけが異なる二状態で同一になる。
- beta の診断文は「発進可能」のような全条件の充足を断定せず、alpha への安全確認が必要であることを表す。
- `hivc_sim/tests` が通過し、再生成した集計 CSV が修正後の実装から作られている。

## Phase 0: 実装前の文書・API確認

### 確認済みの仕様と既存API

| 項目 | 参照 | 実装で使うもの |
|---|---|---|
| 役割別情報の境界 | `REQUIREMENTS.md` §4.6 | alpha は浸水・船体、beta は通信・艇を優先観測し、真の状態を事前に渡さない。 |
| 発進の全条件 | `REQUIREMENTS.md` §4.3 | `flooding <= 3` は勝利判定に残す。ただし beta の私有診断から断定しない。 |
| 時系列指標 | `REQUIREMENTS.md` §6 | `plan_revision_quality` と `route_switch_quality` はゲーム内のターン遷移を評価対象とする。 |
| 行データのゲーム識別子 | `hivc_sim/turn_game_metrics.py` の `terminal_metrics` | `condition` がある場合はそれも含め、`seed` を優先し、なければ `game_id` を使う。条件間で同じ seed を再利用しても結合しない。 |
| beta への入力経路 | `scripts/llm_turn_game_common.py` の `format_state`、`_role_evidence` | プロンプトへ渡す役割情報は `role_specific_evidence("beta", state)` の戻り値だけを修正する。 |

### 禁止事項

- 全バッチを `turn` だけでソートして前状態を保持しない。
- beta 用の診断で `_escape_conditions_met()` / `_escape_conditions_met_current()` の真偽を公開しない。
- `flooding` を beta の状態表示・診断・ログ上の beta 向けプロンプトに追加しない。
- `state_before` / `state_after` の全状態ログを削除しない。これはシミュレータ側の監査ログであり、LLM 入力とは別である。

## Phase 1: 複数ゲーム時系列指標をゲーム単位に分離

### 実装内容

1. `hivc_sim/turn_game_metrics.py` に、既存の `terminal_metrics` と同じ優先順位でゲームを識別する小さな内部ヘルパーを追加する。
   - `condition` が存在すればキーに含める。ゲーム本体は `seed` が存在すればその値、なければ `game_id`、どちらもなければ後方互換の既定値を返す。
2. `plan_revision_quality(rows)` をゲーム識別子ごとにグループ化し、各グループを `turn` 昇順で処理する。
   - `prev_regret` は各ゲームの先頭で `None` に初期化する。
   - 分子・分母は全ゲームの有効なイベント遷移について合算する。
3. `route_switch_quality(rows)` も同じグループ化を適用する。
   - `prev_planned` は各ゲームの先頭で初期化する。
   - 既存の「イベントターンかつ前計画と最適経路が異なる」分母定義は維持する。
4. 公開関数の引数・戻り値・CSV列名は変更しない。

### テスト

`hivc_sim/tests/test_turn_game_metrics.py` に次を追加する。

- 別々の `seed` の turn 0 行を混在させても、ゲーム境界だけを理由に `route_switch_quality` の分母が増えず `NaN` になるケース。
- 同一 `seed` でも `condition` が異なる turn 0 行を混在させ、条件境界が前ターン扱いされないケース。
- 二ゲームの行をインターリーブしても、各ゲーム内の regret 遷移から得る `plan_revision_quality` が期待値どおりになるケース。
- 同じデータをゲーム単位の順序と turn 単位に混ざった順序で渡しても、両指標が等しいケース。

### 検証チェックリスト

- 上記の旧最小再現（独立した2ゲームの turn 0）で `route_switch_quality=1.0` にならない。
- `pytest hivc_sim/tests -q` が通る。
- 指標の docstring を「ゲームごとに時系列処理する」と明記する。

## Phase 2: beta の発進診断を可視情報だけに限定

### 実装内容

1. `hivc_sim/turn_game.py` の `pod_ready_status(state)` を beta 専用の**部分条件診断**に変更する。
   - `_post_start_state(state)` で可視資源へのターン開始時効果を反映する処理は維持する。
   - `pod_readiness`、`pod_integrity`、`oxygen`、`power` の未達は従来どおり列挙する。
   - 可視条件がすべて満たされる場合は、全体の発進可否ではなく、例えば「艇・資源の準備完了（浸水・船体は安全担当に確認）」を返す。
   - `_escape_conditions_met*()` は `step()` と方策の内部判定にのみ残し、beta のテキスト生成では使わない。
2. `role_specific_evidence("beta", state)` のラベルを「発進条件充足」から「beta確認済みの発進準備」などへ変え、全条件を知っているという含意を取り除く。
3. `REQUIREMENTS.md` §4.6 の beta の観測記述を「発進条件」から「通信・脱出艇と可視資源の準備条件」に明確化し、最終発進可否には alpha の安全情報が必要であることを追記する。
4. `scripts/llm_turn_game_common.py` の `format_state` のマスク仕様は維持する。必要であればプロンプト文に「浸水・船体の最終安全確認は alpha へ問い合わせ」と一文追加するが、真値は注入しない。

### テスト

`hivc_sim/tests/test_turn_game.py` に次を追加する。

- `oxygen`、`power`、艇の二状態を固定し、`flooding=0` と `flooding=4` の beta 診断文字列が完全一致するケース。
- 可視条件に不足がある場合だけ、その不足種別（整備・艇損傷・酸素・電力）が診断に含まれるケース。
- `format_state(state, "beta")` が浸水度・船体損傷の数値を含まない既存保証を明示するケース。
- シミュレータ内部では同じ二状態で行動 F の勝敗が異なることを確認し、**ゲーム判定は残しつつ情報公開だけを遮断した**ことを示すケース。

### 検証チェックリスト

- beta のプロンプト文字列を比較して、隠し状態だけの変更で差分が出ない。
- alpha の浸水・船体診断と、`step()` の発進勝敗判定は不変である。
- `pod_ready_status` の説明コメントと要件文が「全体の発進可否を返さない」点で一致する。

## Phase 3: 回帰確認と成果物の更新

1. 全テストを実行する。

```bash
uv run --with pytest --with numpy --with pandas --with scipy --with matplotlib --with tqdm pytest hivc_sim/tests -q
```

2. 既存の policy 実験出力を更新対象としている場合は、`hivc_sim/turn_game_cli.py` の既存コマンドで heuristic / MCTS の CSV と summary を再生成する。修正前の `plan_revision_quality` / `route_switch_quality` を含む summary を残さない。
3. `git diff --check` を実行し、コード・要件・テスト以外の無関係な差分を混入させない。
4. `document/development-log.md` に、修正内容・テスト結果・再生成した成果物を追記する。

## 3. 受け入れ判定

以下をすべて満たした時点で完了とする。

- 2件の回帰テスト群が追加され、修正前の実装では失敗する。
- バッチの行順・ゲーム数を変えても、ゲーム内履歴が同じなら時系列指標は不変である。
- beta のプロンプトと役割固有情報から、隠された浸水・船体状態を二値的に判定できない。
- 全テストが成功し、要件・開発ログ・生成済み集計の整合が取れている。

## 4. 対象外

`scripts/qwen_turn_game_agent_smoke.py` と `scripts/qwen_two_agent_turn_game_smoke.py` は、現時点では単体／旧 smoke 用に全状態を表示する別経路である。本修正では、役割別情報を使う共通二エージェントランナー (`scripts/llm_turn_game_common.py`) の情報境界だけを対象とする。smoke runner を役割別実験へ再利用する場合は、同じマスク仕様への統合を別タスクとして扱う。
