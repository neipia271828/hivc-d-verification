# エージェント・フレームワークプロンプト設計

## 1. 目的

HIVC-D、一般的な意思決定フレームワーク、研究者が任意に記述した手順を、二者エージェントへ明示的に与え、その有無・内容・版を再現可能に比較できるようにする。

ここでいう**フレームワークプロンプト**は、ゲームの勝敗や真の状態を変更するものではなく、エージェントが情報共有・質問・優先順位の調整・最終判断をどの順序で行うかを指示する共有テキストである。

## 2. 背景と設計原則

現行の `control`、`consulting`、`hivc_d` は、実装内の `CONDITION_PROCEDURES` に固定された手順文である。このままでは、次の問題がある。

- 実験者が新しいフレームワークを試すたびにコードを編集する必要がある。
- 実行ログだけでは、実際にどの文面・どの版をエージェントへ渡したか監査できない。
- ペルソナ、ゲーム規則、合意形成手順という異なる介入を同じ「条件」に混在させやすい。

新設計は次の原則に従う。

1. フレームワークは**共有の合意形成手順**であり、alpha/beta の役割・性格を定めるペルソナとは分離する。
2. 比較実験では、フレームワーク以外（ゲーム規則、seed、シナリオ、役割別観測、ペルソナ、生成・議論予算）を同一にする。
3. 任意テキストを受け入れる場合も、実行時に解決された本文・識別子・ハッシュを結果へ保存する。
4. フレームワークはゲーム規則、非公開情報、出力JSON契約、質問応答閉包を上書きできない。
5. 既存の `control`、`consulting`、`hivc_d` は後方互換の組込みプロファイルとして残す。

## 3. 用語と介入境界

| 用語 | 定義 | 例 | 同一比較内での扱い |
|---|---|---|---|
| ゲーム規則 | 状態、行動、勝敗、情報非対称性 | 脱出条件、alpha の浸水観測 | 固定。プロンプトから変更不可 |
| ペルソナ | エージェント固有の役割・優先傾向 | 安全担当、通信担当 | 固定または独立因子として記録 |
| フレームワーク | 二者に共通して与える合意形成の手順 | HIVC-D、リスク最小化手順 | 主たる操作変数 |
| 条件 | 実験結果を集計する比較単位 | `control`、`hivc_d`、`custom_foo` | `framework_id` と対応付ける |
| エージェント追加指示 | 片方だけへ与える任意テキスト | alpha の作業メモ | 主比較とは分離し、明示的な別因子として扱う |

「適当なフレームワーク」は、まず共有フレームワークとして登録する。片方だけに異なる指示を与える実験は可能にするが、HIVC-D 対照実験と同じ条件ラベルや同じ集計群に混在させてはならない。

## 4. フレームワーク・プロファイル

### 4.1 プロファイル形式

フレームワークは YAML または JSON のプロファイルとして管理する。1ファイルには1プロファイルを置き、ファイル本文を実行時の正本とする。

```yaml
id: hivc_d
version: "1.0"
label: "HIVC-D"
scope: shared
description: "I共有、V整合、A確認に基づく合意形成手順"
prompt: |
  以下の順序で合意を形成する。
  1. I（情報）: 観測済みの状態・イベント・資源を共有する。
  2. V（判断基準）: 優先順位の不一致を示し、共通の優先順位を仮定する。
  3. A（能力）: 選択行動が観測済みの制約を満たすか確認する。
```

必須項目は `id`、`version`、`label`、`scope`、`prompt` とする。

- `id` は英小文字、数字、`_`、`-` からなる一意の識別子とする。
- `scope` は初期実装では `shared` のみを許可する。
- `prompt` は空文字を許可し、空文字は追加手順なしの対照を表す。
- `version` を変更せずに `prompt` を変更してはならない。

### 4.2 組込みプロファイル

次の3プロファイルを提供する。組込み版も外部プロファイルと同じスキーマへ正規化してログへ出力する。

| id | label | 目的 |
|---|---|---|
| `control` | Control | 追加の合意形成手順なし |
| `consulting` | General consulting | 一般的なリスク比較・根拠確認の助言 |
| `hivc_d` | HIVC-D | I共有、V整合、A確認の順序を与える |

### 4.3 任意フレームワーク

実験者は次のいずれかで任意のフレームワークを与えられる。

1. `frameworks/<id>.yaml` のプロファイルファイルを指定する。
2. CLIまたは実験設定でプロファイルファイルへのパスを指定する。
3. 単発の探索実験に限り、UTF-8テキストファイルを `prompt_file` として指定し、実行時に一時プロファイルへ正規化する。

CLI上の長い文字列引数で直接本文を渡す方式は、シェル履歴への残留・引用符の破損・再現性低下を招くため、正規経路にしない。

## 5. 入力インターフェース

### 5.1 実験設定

`configs/experiment.yaml` は既存の `conditions` を維持しつつ、次の新しい指定を受け付ける。

```yaml
# 既存の条件指定（組込みプロファイルとの後方互換）
conditions: [control, hivc_d]

# 外部プロファイルを含む比較実験
framework_profiles:
  - frameworks/control.yaml
  - frameworks/hivc_d.yaml
  - frameworks/risk_matrix_v1.yaml

# framework_profiles を使わない単発探索用
framework_prompt_file: null
```

- `framework_profiles` を指定した場合は、そこに列挙したプロファイルを実験単位として実行する。
- `conditions` と `framework_profiles` を同時指定した場合はエラーとする。どちらの操作変数か曖昧にしないためである。
- `framework_prompt_file` は単発実験専用とし、複数プロファイル比較との併用を禁止する。
- 外部プロファイルの `id` は、同一run内および組込みプロファイルと重複してはならない。

### 5.2 CLI

実験ランナーには次を追加する。

```text
--framework-profiles <path> [<path> ...]
--framework-prompt-file <path>
```

設定ファイルの値はCLI指定で上書きできる。最終的に解決されたプロファイル一覧を実験開始時に標準出力へ表示する。

### 5.3 将来のエージェント個別指示

将来、alpha/beta に別々の追加指示を渡す場合は、`agent_prompt_overrides` を別設定として導入する。その際は以下を必須とする。

- `agent_prompt_condition_id` を通常の `framework_id` と別列で記録する。
- 共有フレームワークとの同時指定・優先順位を明記する。
- 個別指示ありのrunを、共有フレームワークのみの主比較結果と混在集計しない。

本要件の初期実装対象は共有フレームワークに限定する。

## 6. プロンプト合成と優先順位

各エージェントの自由議論プロンプトと意思決定プロンプトには、同じフレームワークブロックを挿入する。自由議論だけに与え、投票時に消える設計は採用しない。

プロンプトの優先順位は以下とする。

```text
システム実装上の不変ルール
  > ゲーム規則・役割別観測・出力JSON契約・質問応答閉包
  > 共有フレームワークプロンプト
  > ペルソナ
  > 過去の議論
```

フレームワーク本文は明確な区切りで挿入し、次の命令は無効として扱う。

- 非公開の状態値や探索結果を開示させる命令
- 行動一覧、勝敗条件、役割別観測を変更する命令
- JSON形式、投票方式、質問応答閉包、予算上限を無視させる命令
- 他のプロファイル、過去run、評価用正解を参照させる命令

この検証は完全な意味理解を要求しない。初期実装では危険な明示キーワードと構造を拒否し、実行プロンプトにも「上位のゲーム規則・出力契約を変更しない」と明記する。拒否されたプロファイルは実験を開始せず、理由を表示する。

## 7. ログ・再現性・プレビュー

### 7.1 run メタデータ

各実験出力ディレクトリに `framework_manifest.json` を保存する。少なくとも次を含める。

- `framework_id`、`label`、`version`、`scope`
- 実行時に解決した `prompt` の完全なUTF-8本文
- 本文の SHA-256
- プロファイルの入力パスまたは `builtin` / `prompt_file` の出所
- 実行時刻、実験設定ファイル、ランナーのGit commit
- 使用したペルソナ指定、seed範囲、ゲーム数

### 7.2 ターン行と集計

すべてのターンCSV行に次を追加する。

```text
framework_id, framework_label, framework_version, framework_prompt_sha256
```

`condition` は後方互換のため残すが、外部プロファイル実験では `framework_id` と同じ値に正規化する。集計・ゲーム識別・CSV結合は `framework_id` を優先し、同一seedでも別フレームワークのターン履歴を混在させない。

### 7.3 ローカルプレビュー

ローカルプレビューには以下を追加する。

- フレームワークID・ラベル・版・SHA-256の短縮表示
- `framework_manifest.json` の全文プロンプトを展開表示するパネル
- 条件タブを `framework_id` ごとに表示する
- 同じゲームseed・turnを選んだ複数フレームワークの会話・投票・評価を比較できる表示

プレビューに表示する本文は、CSVの推定値ではなくmanifestに保存された実行時本文を正本とする。

## 8. 実験妥当性

### 8.1 主比較

HIVC-D の効果を評価する主比較は、次の3プロファイルとする。

| framework_id | 追加手順 | 位置付け |
|---|---|---|
| `control` | なし | 負の対照 |
| `consulting` | 一般的なリスク管理・根拠確認 | 非特異的な助言対照 |
| `hivc_d` | I共有、V整合、A確認 | 検証対象 |

各プロファイルで、同一のモデル版、生成設定、seed、scenario_id、役割別観測、ペルソナを使う。任意フレームワークは探索的比較として同じ表に追加できるが、事前登録済みの主仮説と事後探索を区別して報告する。

### 8.2 禁止事項

- フレームワークごとに別のゲーム規則、乱数seed、役割別情報、探索評価値を与えない。
- フレームワーク本文に特定シナリオの正解行動や事前ロールアウトの個別結果を含めない。
- 同じ `framework_id` の本文・版をrun途中で変更しない。
- 本文・ハッシュ・出所を保存せずに結果CSVだけを比較しない。
- ペルソナ変更をフレームワーク効果として報告しない。

## 9. 受け入れ基準

1. 組込みの `control`、`consulting`、`hivc_d` を、外部プロファイルと同じ内部表現で実行できる。
2. YAMLプロファイルを追加するだけで、コード編集なしに新しい共有フレームワークを実行できる。
3. 同一のプロファイル・モデル・seed・設定から、同一の `framework_prompt_sha256` を持つrunを再実行できる。
4. 自由議論・意思決定機会の両方に、同一フレームワーク本文が与えられる。
5. フレームワークが空でも、ゲーム規則、出力JSON、質問応答閉包、フォールバック規則が現行と同じに動く。
6. 無効なID、重複ID、必須項目欠落、危険な上書き要求を含むプロファイルは実行前に明確なエラーとなる。
7. CSV、`summary.csv`、`framework_manifest.json`、ローカルプレビューで、表示中の結果に対応するフレームワーク本文と版を追跡できる。
8. 既存の `conditions` 指定だけで動くコマンドと既存のrunプレビューを壊さない。

## 10. 実装順序（未実施）

1. プロファイルの読込・検証・正規化を担うモジュールを追加し、組込み手順を移行する。
2. 実験設定・CLI・ランナーへ `framework_profiles` と `framework_prompt_file` を追加する。
3. 共通プロンプト生成をプロファイル入力へ置換し、自由議論・意思決定双方へ挿入する。
4. 行CSV、summary、manifestへフレームワーク来歴を記録する。
5. ローカルプレビューにmanifestとフレームワーク比較表示を追加する。
6. プロファイル検証、後方互換、CSVスキーマ、同一seed比較、プレビュー表示のテストを追加する。
7. GPUで `control` / `consulting` / `hivc_d` 各1ゲームと、任意プロファイル1ゲームを実行し、取得済みログのプレビューで本文・ハッシュ・会話を確認する。

## 11. 対象外

- エージェントが任意フレームワークを自動選択・自動生成する機能
- フレームワーク本文の効果を事前に真偽判定する機能
- ゲーム規則を自然言語プロンプトから変更する機能
- 複数フレームワークを1ターン内で動的に切り替える機能
- ペルソナ設計・ゲームシナリオ設計そのものの変更
